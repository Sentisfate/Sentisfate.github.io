关于package.json

2023-03-09 03:19

name

表示包的名字注意事项

  • 名称可以通过require引入
  • 不能与其他模块名称重复，可以通过 npm view name 来查看是否重复，不重复报404(如果不需要发布到npm上，则只需保证与本地的包不同就行)

version

标识版本号，版本号的命名遵循语义化版本 2.0.0 规范，格式为：「主版本号. 次版本号. 修订号」，通常情况下，修改主版本号是做了大的功能性的改动，修改次版本号是新增了新功能，修改修订号就是修复了一些 bug.如果某个版本的改动较大，并且不稳定，可能如法满足预期的兼容性需求，就需要发布先行版本，先行版本通过会加在版本号的后面，通过 “-” 号连接以点分隔的标识符和版本编译信息：内部版本（alpha）、公测版本（beta）和候选版本（rc，即 release candiate）查看react所有版本 npm view react versions查看react最新版本 npm view react version

description

一个字符串，可以让其他开发者在npm的搜索中发现项目包

"description": "详细解释package.json中的各个字段，教你如何在npm上发布一个包",

keywords

字符串数组，增加项目包的曝光率

"keywords": ["package.json字段详解","npm","javascript"],

author

作者，有两种写法

"author": "不讲码德 <1135643799@qq.com> (https://www.yuque.com/antusheng-2wzx0)",

"author":{
  "name":"不讲码德",
  "email": "1135643799@qq.com",
  "url": "https://www.yuque.com/antusheng-2wzx0"
},

contributors

是一个数组，每个元素的写法同author一样，有两种写法

homepage

项目主页的地址，一个url字符串

"homepage": "https://www.yuque.com/g/antusheng-2wzx0/awgkrd/msvwxozpbdkbfiqp/collaborator/join?token=YkPmNoBGPFudUJfe# 《关于package.json》",

repository

仓库地址

"repository": {
  "type": "git",
  "url": "https://gitee.com/ssttddeeff/npm-package-study",
  "directory": ""
},

bugs

表示提交问题的地址，为一个对象，可以添加一个提交问题的地址和反馈邮箱。

"bugs": {
  "email": "1135643799@qq.com",
  "url": "https://www.yuque.com/g/antusheng-2wzx0/awgkrd/msvwxozpbdkbfiqp/collaborator/join?token=YkPmNoBGPFudUJfe# 《关于package.json》"
},

dependencies

生产环境依赖，指的是，项目打包上线后仍需要用到的依赖。当我们用 npm i package时，默认就会在生产环境，这行命令是npm install --save package的简写。执行npm i dayjs后

"dependencies": {
  "dayjs": "^1.11.7"
}

该对象分别由包名和版本组成，版本号遵循**「主版本号. 次版本号. 修订号」**的原则。

  • 固定版本： 1.11.7
  • 波浪号：比如~1.11.7 表示安装1.11.x的最新版，不低于1.11.7
  • 插入号： ^1.11.7 表示安装不低于1.11.7的1.x.x版本

💡当手动修改了依赖里面的配置后，执行**npm i **会安装相应的版本和包

devDependencies

开发环境依赖，在进行项目打包部署上线时，其中的依赖不会被打包进去，只会在本地开发环境存在。npm i --save-dev 包名简写为npm i -D 包名

  "devDependencies": {
    "nodemon": "^2.0.21"
  }

peerDependencies

peerDependencies是为了解决插件所依赖的包与宿主环境所依赖的包不一致的问题。执行npm i antd后，打开node_modules下的antd文件夹里的package.json， 

  "peerDependencies": {
    "react": ">=16.9.0",
    "react-dom": ">=16.9.0"
  },

能看到antd这个模块依赖的是react和react-dom这两个模块的版本。它要求宿主环境需要安装对应的版本的包。这样才能保证包的一致性。

optionalDependencies

如果需要在找不到包，或者包安装失败时，npm仍能继续运行，可以将包放在optionalDependencies里。这种情况要对optionalDependencies的包做异常处理

bundledDependencies

打包依赖，值为一个数组，在发布包时，里面的依赖都会被打包。

"bundledDependencies":["nodemon"],

执行npm pack 后，压缩包里就会多一个node_modules文件夹，以及里面对应的包

engines

用于说明项目包对npm和node的版本要求。只是一个提示说明，不会影响依赖的安装。

"engines": {
 "node": ">=8.10.3",
  "npm": ">=6.9.0"
}

scripts

配置脚本，是一个对象，包含一系列键值，key为命令，value 为实际的命令操作，可以使用npm run key来执行value

  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "start": "nodemon index.js"
  },

同时，可以结合pre和post完成前置，和后置操作。

  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "start": "node index.js",
    "prestart": "node preIndex.js",
    "poststart": "node postIndex.js"
  },

console.log("前置脚本....");

console.log("postdev....");

config

用来配置scripts运行时的配置参数，里面的字段会被映射到process.env.npm_package_config_字段名

  "config": {
    "port":3000
  }

console.log("postdev....");
console.log(process.env);
console.log(process.env.npm_package_config_port)

main

用来指定加载的入口文件，浏览器和node环境都可以使用。require(name)的返回就是main指定的文件的module.exports属性。

"main": "index.js",

browser

定义npm包在browser环境下的入口文件，如果npm包只在Web端使用，禁止在服务端使用，使用browser来定义入口文件

"browser":"index.js"

module

module用于定义npm包ESM规范的入口文件，包是ESM规范的，使用module来定义

"module": "./src/index.mjs",

注意：在 Web 环境中，如果使用 loader 加载 ESM（ES module），那么这三个配置的加载顺序是 browser→module→main，如果使用 require 加载 CommonJS 模块，则加载的顺序为 main→module→browser。

bin

很多软件包都有一个或多个可执行文件，使用bin字段可以将其安装到PATH中。例如：

  "bin": {
    "myapp":"./bin/test-cli.js"
  },

当包被全局安装时，该可执行文件会被添加进PATH并且可以被通过"myapp"命令运行。本地写一个简单的包，在根目录下创建bin文件夹，里面写一个test-cli.js，并在package.json里配置好。

#!/usr/bin/env node
// 上面这行代码告诉系统要通过node来执行下面的代码

console.log("hello world");

执行 npm link将本地包link到全局或者npm install . -g(如果有更改，需要npm publish来更新，推荐使用前者)安装到全局，能看到全局node_modules下多了一个本地的包链接。

打开终端执行 myapp命令

同时，可以还在script里配置可以运行的脚本

"scripts": {
  "test": "echo \"Error: no test specified\" && exit 1",
  "start": "node index.js",
  "prestart": "node preIndex.js",
  "poststart": "node postIndex.js",
  "hello":"myapp"
},
"bin": {
  "myapp":"./bin/test-cli.js"
},

执行 npm run hello 本质上是一样的。更多参看：node 命令行接口 https://www.npmjs.com/package/commander

files

files 配置一个数组，用来描述当npm 包作为依赖包安装时需要说明的文件列表。npm包发布时，files指定的文件会被推送到npm服务器里，指定的是文件夹，里面的内容都会被推送。

  "files": [
    "index.js"
  ]

如果有不想提交的文件，可以在根目录里新建一个.npmignore，类似.gitignore

node_modules
.vscode
build
*.tgz

man

man 是linux的帮助指令，通过指令可以查看Linux的指令帮助，配置文件帮助和编程帮助等信息。

"man": [
  "./man/npm-access.1",
  "./man/npm-audit.1"
]

需要注意：

  • man 文件必须以数字结尾，如果经过压缩，还可以使用. gz 后缀。这个数字表示文件安装到哪个 man 节中；
  • 如果 man 文件名称不是以模块名称开头的，安装的时候会加上模块名称前缀。
man npm-access
man npm-audit

private

用于防止将私有库发布到npm服务器上

"private":false

preferGlobal

表示当前包更倾向于全局模块，如果用户没用以全局方式安装，在设置为true时，会发出警告。

publishConfig

publishConfig 配置会在模块发布时生效，用于设置发布时一些配置项的集合。如果不想模块被默认标记为最新，或者不想发布到公共仓库，可以在这里配置 tag 或仓库地址。

"publishConfig": {
  "tag": "1.1.0",
  "registry": "https://registry.npmjs.org/",
  "access": "public"
}

os

设置npm包应该在什么操作系统使用

"os" ["linux"]   // 适用的操作系统
"os" ["!win32"]  // 禁用的操作系统

cpu

"cpu" ["x64", "AMD64"]   // 适用的cpu
"cpu" ["!arm", "!mips"]  // 禁用的cpu

license

开源协议

  • MIT 只要用户在项目副本中包含了版权声明和许可声明，他们就可以拿你的代码做任何想做的事情，你也无需承担任何责任。
  • Apache 类似MIT
  • GPL 修改项目代码的用户再次分发源码或二进制代码时，必须公布他的相关修改。
"license": "MIT"
